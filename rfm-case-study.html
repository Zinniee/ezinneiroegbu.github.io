RFM Customer Segmentation for a Mobility Startup

Using RFM analysis with Python, MySQL, and Tableau to drive targeted marketing, improve retention, and grow B2C acquisition for a tech-enabled transport platform.

Role Data Analyst (Individual Contributor)
Company Shuttlers (Nigeria)
Period Oct 2021 – Jun 2022
Python MySQL Tableau RFM Analysis Customer Segmentation
Note: Visuals and code snippets are sanitized to protect confidential information while demonstrating approach and impact.

1. Overview

Shuttlers is a mobility startup providing tech-enabled transportation services. The business wanted to move beyond a “one size fits all” approach to B2C riders and start treating customers differently based on their value and engagement.

I led the design and implementation of an RFM customer segmentation model and an accompanying Tableau dashboard. This allowed marketing and growth teams to see exactly which riders were most valuable, at-risk, or inactive – and to tailor campaigns accordingly.

Headline impact: Targeted insights from the RFM model contributed to a 34.75% increase in B2C user acquisition in H1 2022 and improved focus on the most profitable customer segments.

2. Business Problem

Before this project, all B2C riders were treated as a single group:

  • No structured definition of “high value” vs “low value” riders.
  • Marketing campaigns were broad and not tailored to behaviour.
  • There was no easy way to see which customers were reactivatable vs truly churned.

The core problem we wanted to solve: “How can we use transaction and ride data to understand customer value and prioritize marketing efforts?”

3. Data Sources & Architecture

I worked with data from three core systems:

  • Ride / Trip data – ride ID, customer ID, route, fare amount, trip date.
  • Customer data – sign-up date, contact information, segment (B2B vs B2C).
  • Payment data – transaction amount, timestamps, payment method.

All operational data landed in MySQL. For the RFM model, I:

  • Wrote SQL queries to join and aggregate customer-level metrics.
  • Exported or connected to the data via Python for scoring logic.
  • Pushed the final RFM-scored table to Tableau for visualization.
Key design decision: Keep the RFM calculations in a reproducible Python/SQL workflow so it could be refreshed on a regular cadence without manual Excel work.

4. RFM Methodology

I implemented a standard RFM framework, tailored to the ride context:

  • Recency (R): days since last completed ride.
  • Frequency (F): number of completed rides over the last 6–12 months.
  • Monetary (M): total fare amount spent over the same period.

First, I built a base aggregation in SQL:

-- Aggregate RFM base table (simplified) SELECT c.customer_id, MAX(t.trip_date) AS last_trip_date, COUNT(*) AS ride_count, SUM(t.fare_amount) AS total_spend FROM trips t JOIN customers c ON c.customer_id = t.customer_id WHERE t.status = 'completed' AND t.trip_date >= DATE_SUB(CURDATE(), INTERVAL 12 MONTH) GROUP BY c.customer_id;

Then, in Python, I calculated RFM scores and converted them into segments:

import pandas as pd from datetime import date df = rfm_base_df.copy() today = date.today() df["recency_days"] = (today - df["last_trip_date"]).dt.days # Assign R, F, M scores 1-5 using quantiles (example) df["R_score"] = pd.qcut(df["recency_days"], 5, labels=[5,4,3,2,1]).astype(int) df["F_score"] = pd.qcut(df["ride_count"], 5, labels=[1,2,3,4,5]).astype(int) df["M_score"] = pd.qcut(df["total_spend"], 5, labels=[1,2,3,4,5]).astype(int) df["RFM_score"] = df["R_score"] + df["F_score"] + df["M_score"] def assign_segment(row): if row["RFM_score"] >= 13: return "Champions" if row["RFM_score"] >= 10: return "Loyal" if row["R_score"] >= 4 and row["F_score"] <= 2: return "Promising" if row["R_score"] <= 2 and row["F_score"] >= 3: return "At Risk" return "Low Value" df["segment"] = df.apply(assign_segment, axis=1)

The exact thresholds were tuned iteratively with marketing and product teams to ensure that segments were both statistically meaningful and business-meaningful.

5. Solution Design – Tableau Dashboard

Once the RFM table and segments were created, I designed a Tableau dashboard with three layers:

  • Executive overview: total customers by segment, revenue contribution per segment, and trend of active vs inactive riders.
  • Segment drill-down: ability to filter by segment (e.g. “Champions”, “At Risk”) and see geo distribution, route usage, and fare patterns.
  • Customer-level view: table of riders with their RFM scores, last ride date, and contact information for export into campaign tools.
Example dashboard interactions:
  • Filter to “At Risk” riders with high past monetary value.
  • Export their contact details for a reactivation campaign.
  • Track uplift in rides and revenue after the campaign.

7. Challenges & How I Addressed Them

1. Data quality & missing values

Some customers had very few rides or inconsistent data. I:

  • Set minimum activity thresholds (e.g. at least 2 completed rides) for RFM inclusion.
  • Flagged suspicious records (e.g. negative fares, test accounts) and excluded them.

2. Choosing the right time window

The business was seasonal in some corridors, so choosing a lookback window was important. I:

  • Tested 6-, 9-, and 12-month windows and compared stability of segments.
  • Aligned with stakeholders on a 12-month window to capture both frequent and seasonal riders.

3. Explaining RFM to non-technical stakeholders

RFM can feel abstract if you’re not used to it. To address this, I:

  • Created a simple explainer slide with real customer examples per segment.
  • Linked each segment to a proposed marketing action (e.g. upsell, reactivate, reward).

8. What I’d Improve Next

If I had more time and resources, I would extend this work in a few ways:

  • Add churn prediction: build a simple model using RFM + behaviour features (routes, days of week) to predict likelihood of churn.
  • Test campaign uplift formally: run A/B tests comparing RFM-based targeting vs generic campaigns.
  • Automate refresh: schedule a weekly RFM pipeline so segments are always up-to-date for marketing and customer success teams.

Overall, this project solidified my approach to turning raw transaction data into a marketing and growth decision tool that non-technical teams could actually use.
