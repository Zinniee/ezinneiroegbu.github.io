SLA & Complaints Performance Dashboard

Using Power BI, DAX, Python and SQL to monitor complaints, reduce SLA breaches to single digits, and improve customer experience across banking channels.

Role Data Analyst (Individual Contributor)
Company Sterling Bank Plc, Nigeria
Period Jun 2022 – Jul 2024
Power BI DAX Python SQL / MySQL SLA Monitoring Customer Complaints Analytics
Note: Visuals and code snippets are sanitized to protect confidential information while demonstrating approach and impact.

1. Overview

At Sterling Bank Plc, customer complaints and SLA performance were monitored through multiple static reports and spreadsheets. This made it hard for leadership to quickly see:

  • How many complaints were being raised across channels and products.
  • Which teams were breaching SLA targets and why.
  • How complaint resolution performance impacted customer experience and the bottom line.

I designed and built a centralized SLA & Complaints Performance Dashboard in Power BI, backed by DAX measures and Python/SQL analytics. It became the single source of truth for tracking complaint volumes, SLA adherence, and root causes.

Headline impact: the new dashboard helped reduce complaints SLA breaches to single digits and consistently maintain customer experience SLAs at around 92%+.

2. Business Problem

Before the dashboard, the complaints management process faced several challenges:

  • Fragmented reporting: different teams maintained their own complaint logs, making it hard to get a consolidated view.
  • Manual tracking: SLA performance was tracked manually in Excel, which was slow and error-prone.
  • No clear ownership: it was difficult to quickly identify which unit/channel was driving SLA breaches.
  • Limited insight: reports showed counts, but not trends, patterns, or root causes.

The goal was to build a live, self-service dashboard that answered questions like: “Which channels are driving most complaints?”, “Where are we breaching SLAs?”, and “How is our SLA performance trending month-over-month?”

3. Data Sources & Data Model

I worked with several key data sources:

  • Complaint tickets data – ticket ID, customer ID, product, channel, opened date, resolved date, status, SLA target, resolution category.
  • Customer & product reference data – mapping to segment, region, product type.
  • Service measurement metrics – customer satisfaction scores, NPS-like metrics, and internal performance KPIs.

The data was stored in relational tables (SQL Server / MySQL). I modelled it in Power BI using a star schema approach:

  • FactComplaints – one row per complaint ticket with timestamps and SLA info.
  • DimCustomer – customer segment, region, etc.
  • DimProduct – product type, product group.
  • DimChannel – branch, mobile app, contact centre, etc.
  • DimDate – standard calendar table for time intelligence.
Key decision: keep the model lean and analytics-ready by separating complaint facts from descriptive dimensions and building reusable DAX measures on top.

4. Core Metrics & DAX Logic

I worked with stakeholders to define the core metrics that really mattered to them:

  • Total Complaints – number of complaint tickets in a period.
  • Within SLA – tickets closed on or before SLA due date.
  • SLA Breaches – tickets closed after SLA due date or still open past SLA.
  • % SLA Met – percentage of complaints resolved within SLA.
  • Avg Resolution Time – average hours/days to close a complaint.

Example DAX measures (simplified):

-- Total Complaints Total Complaints := COUNTROWS ( 'FactComplaints' ) -- Complaints Resolved Within SLA Complaints Within SLA := CALCULATE ( [Total Complaints], 'FactComplaints'[ResolvedDate] <= 'FactComplaints'[SLA_DueDate], 'FactComplaints'[Status] = "Closed" ) -- SLA Breaches SLA Breaches := CALCULATE ( [Total Complaints], 'FactComplaints'[ResolvedDate] > 'FactComplaints'[SLA_DueDate] || ( 'FactComplaints'[Status] <> "Closed" && TODAY() > 'FactComplaints'[SLA_DueDate] ) ) -- SLA Met % SLA Met % := DIVIDE ( [Complaints Within SLA], [Total Complaints] ) -- Average Resolution Time (in hours) Avg Resolution Time (hrs) := AVERAGEX ( 'FactComplaints', DATEDIFF ( 'FactComplaints'[OpenedDate], 'FactComplaints'[ResolvedDate], HOUR ) )

For deeper analysis, I also used Python and SQL to:

  • Identify patterns in complaint reasons and repeat offenders.
  • Group complaints by severity and channel.
  • Generate distributions and trend lines used in Power BI visuals.

4a. SQL Aggregation (Sanitized)

Below is a representative query structure used to produce a ticket-level fact table and SLA indicators. Table/column names are generalized.

-- SLA-ready complaint fact table (sanitized) WITH base AS ( SELECT ticket_id, customer_id, channel, product, opened_at, resolved_at, status, sla_hours FROM complaints WHERE opened_at >= DATEADD(month, -12, GETDATE()) ), derived AS ( SELECT ticket_id, customer_id, channel, product, opened_at, resolved_at, status, DATEADD(hour, sla_hours, opened_at) AS sla_due_at, DATEDIFF(hour, opened_at, COALESCE(resolved_at, GETDATE())) AS age_hours FROM base ) SELECT *, CASE WHEN status = 'Closed' AND resolved_at <= sla_due_at THEN 1 ELSE 0 END AS within_sla_flag, CASE WHEN status = 'Closed' AND resolved_at > sla_due_at THEN 1 WHEN status <> 'Closed' AND GETDATE() > sla_due_at THEN 1 ELSE 0 END AS breach_flag FROM derived;

5. Dashboard Design & User Experience

I designed the Power BI dashboard with three main layers:

  • Executive Summary – SLA Met %, SLA Breaches, Total Complaints, broken down by month, region, and channel. This gave leadership a quick pulse of performance.
  • Operations View – detailed breakdown by product, channel, team/owner, and complaint type. It allowed operations managers to zoom into problem areas.
  • Root Cause & Trend Analysis – category-wise trends, repeated complaint drivers, and customer segment impact.
Examples of filters and interactions:
  • Filter to “Mobile App” channel and see SLA performance by month.
  • Drill down into “Card” complaints and identify the top 3 root causes.
  • Compare SLA Met % between branches and digital channels.

7. Challenges & How I Addressed Them

1. Inconsistent timestamps & missing SLA fields

Some complaints had missing SLA due dates or incorrect timestamps. I:

  • Applied business rules to infer SLA where possible (e.g. product-level SLAs).
  • Flagged records with unreliable data and excluded them from SLA % but included them in volume tracking.

2. Different SLA rules across products

Products like cards, loans, and digital channels had different SLA rules. I:

  • Worked with stakeholders to codify SLA rules per product/channel.
  • Implemented a rules table (Product SLA Matrix) referenced in DAX to compute SLA due dates more consistently.

3. Driving adoption

A new dashboard is only useful if teams use it. To encourage adoption, I:

  • Ran short training sessions for operations and customer experience teams.
  • Created a one-page “how to read this dashboard” guide.
  • Iterated on the layout based on feedback to keep the most-used views easy to access.

8. Future Enhancements

If I extended this project further, I would:

  • Incorporate predictive alerts: use historical SLA breaches to predict weeks or products at higher risk and trigger early warnings.
  • Integrate with case management tools: embed Power BI views directly into the complaint handling system to give agents live context.
  • Link financial impact: quantify the cost of SLA breaches and show a direct link between improved SLA and revenue retention.

Overall, this project showcased my ability to move from raw operational data to a trusted, decision-driving SLA dashboard that supports both strategic and day-to-day decisions.
